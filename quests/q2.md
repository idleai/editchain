Please find large cc session histories at [inputs](../inputs/), those are for the kernel hillclimbing from [ambientlight/rtx-pro-6000-bench](../../rtx-pro-6000-bench/). You should see ~150Mb overall that should make for a reasonably sized editchain. Please make sure we can import this into the edit chain of this scale and then we should focus the effort on implementing a query plane that would allow fast retrieval over the edit chain of such size. 

We can plan for an architecture with hybrid full text (BM25 or equiv) / vector search. Suggest what sensible approach can be used to enable it from our rust codebase. We would want to expose a clean CLI inferface for retrieval, summarization and search. Let's not consider any semantic extraction and this point and lets just get a decent hybrid search in place. Keep in mind our edit chains can get very long - this would need to scale properly to longer DAGs, but consider everything as in-memory. There should be some additional algorithms that can take adventage of our DAG datastructure.

Also, this needs to run fully online as subagents runnings in parallel should expect sub-second consistency on hydrated data from newly added nodes in the chain. This would allow them to be up-to-date as needed with any parallel in progress work.

At first we would use it through a skill that would hit the editchain cli directly (we can experiment with the MCP server on top of editchain but that will be way later).

Also think of few options for the open weights embedding models for our stack.
