## Key design decisions

1. **Synthetic data first** — generator produces realistic chains with branches/merges before connecting to real storage.
2. **No background thread yet** — synchronous loading for the spike; worker thread comes in M1.
3. **Custom DAG lane algorithm** — not Petgraph; `LaneState::compute_rows()` processes newest-first for parent expansion, allocates lanes for branches, handles two-parent merges.
4. **Virtualized rows** — only visible rows rendered via scroll offset; no widget-per-operation.
5. **No RoaringBitmap yet** — simple Vec/HashMap indexes for MVP; RoaringBitmap optimization in M4.
6. **Compact OpHeader** — stores only what's needed for list rendering; full payloads decoded lazily in M3.

---

## DAG lane algorithm

The lane rendering follows the git-log-style approach from the research plan:

1. Process operations **newest-first** for parent expansion
2. Track active lanes: each slot holds the OpId currently occupying that column
3. For each operation:
   - Find or allocate its lane
   - Remove it from the lane
   - Insert parent(s): single parent stays in same lane; second parent gets a new lane (merge)
4. Build rows oldest-first with cell assignments per column position

LaneCell variants: `Empty`, `Vertical`, `Node`, `SelectedNode`, `Horizontal`, `BranchLeft`, `BranchRight`, `MergeLeft`, `MergeRight`, `Crossing`, `ManyParents`

## Synthetic data generator

Produces realistic chains with:
- **Operation kind distribution**: Message (35%), Tool (20%), Command (15%), File (15%), Reflection (5%), Note (5%), Error (3%), Actor (2%)
- **Branches**: ~15% chance of spawning a new branch tip
- **Merges**: ~8% chance of two-parent merge when ≥2 branches exist
- **Deterministic PRNG**: same count produces identical output
- **Preview text**: kind-specific templates (8 variants per kind)
- **Tags**: correct tag bits per kind (AGENT, HUMAN, FILE, TOOL, etc.)
- **Clock**: monotonically increasing Unix ms timestamps
