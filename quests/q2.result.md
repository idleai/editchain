## Architecture

### New crates (3)

```
editchain-query    request/response types, RRF fusion, graph algorithms
editchain-index    Tantivy BM25, flat f16 vector index, chunker, snapshots
editchain-embed    Embedder trait, Qwen3 model manifests
```

### Existing crates modified (3)

```
editchain-codec    EC03 framed format with CRC32C checksums
editchain-import   ID collision fix, blob spill, source lanes, streaming reader
editchain-node     CLI (search/retrieve/summarize/tail), daemon module
```

### Key design decisions

1. **Tantivy in RamDirectory** — no fsync overhead; rebuild from chain on restart.
2. **Exact flat f16 vector index first** — HNSW only after measured latency requires it.
3. **f16 vectors** — half precision halves memory vs f32 with negligible accuracy loss.
4. **RRF fusion** — avoids mixing incompatible BM25 and cosine scores.
5. **ArcSwap/RwLock snapshots** — lock-free reader access; atomic publication.
6. **Unix socket daemon** — warm process keeps indexes + model ready; CLI stays clean.
7. **Qwen3-Embedding-0.6B at 1024d** — served via SGLang on port 8001; HTTP backend uses `curl` subprocess with temp files (Rust HTTP clients hang on this system).

## Import correctness fixes (commit 1)

### ID collision
Replaced `seq * 1000 + 1` with `SourcePosition` packing (48-bit record ordinal + 16-bit derived ordinal shifted left by 16). Added `IdError::Overflow` for explicit error handling.

### Blob spill
Extended `BlobSink` trait with `put()` returning `BlobRef`. Added `payload_for()` helper and `ContentAddressedBlobSink`. Records >4 KiB are spilled to content-addressed blobs instead of truncated.

### Deterministic source lanes
Added `derive_source_stream()` that hashes workspace + session file path for per-file deterministic lanes. Import uses per-file sequence numbering from cursor state.

### Streaming reader
Replaced full-file `read_to_end` with `BufRead::read_until(b'\n')` streaming. Memory per read is now one line at a time.

### EC03 frame format
Added checksummed frame format with CRC32C header + payload checksums alongside EC02 read compatibility. Includes format detection helper.

## Query plane

### Lexical index (Tantivy BM25)

16-field schema with:
- `body`, `body_code` — prose and code-aware text fields
- `kind`, `role` — structured filtering
- `actor_id`, `session_id`, `node_id`, `boot`, `seq` — scope and lane filtering
- `clock_ms`, `generation` — time and consistency filtering

BooleanQuery combines full-text search with kind/actor/session filters.

### Vector index (flat f16)

Row-major f16 vectors in active + sealed segments. RoaringBitmap filters for kind and session pre-filtering before dot product scan. Memory estimate: ~98 MiB for 100K vectors at 512d f16.

### Hybrid fusion

Reciprocal Rank Fusion with default k=60. Default candidates: BM25=200, vector=200 → fused top_k=20.

### DAG-aware features

- **FrontierMap** — fast visibility checks without full graph traversal
- **CausalCone** — ancestor/descendant expansion around seed hits
- **CausalCorridor** — shortest parent path from evidence to tip (`retrieve --why`)
- **MMR diversity rerank** — graph-aware Maximal Marginal Relevance penalty
- **Occurrence collapse** — group duplicate chunks by text similarity

#### Bug Found & Fixed: UTF-8 Boundary in Chunked Text Indexing

**Location**: `crates/editchain-index/src/lexical.rs:138`

**Symptoms**: `editchain search` panicked with:
```
start byte index 10752 is not a char boundary; it is inside '—' (bytes 10751..10754)
```

**Root cause**: The chunker (`chunk_text`) computes byte offsets using a rough token estimate (4 bytes/token), which can land in the middle of multi-byte UTF-8 characters (e.g., em-dash `—` is 3 bytes). When the lexical indexer slices `&text[start..end]` using these offsets, it hits non-char boundaries.

**Fix**: Added `floor_char_boundary()` calls to adjust both start and end offsets to valid UTF-8 boundaries before slicing:

```rust
let start = text.floor_char_boundary(start);
let end = text.floor_char_boundary(end);
let ct = &text[start..end];
```

Also fixed a Rust precedence issue (`as usize.min()` → `(as usize).min()`).

**Also fixed**: The search results had empty `text` fields because the stored Tantivy body field wasn't being extracted from the document. Added:
```rust
let body_text = doc.get_first(self.fields.body)
    .and_then(|v| v.as_str())
    .unwrap_or("")
    .to_string();
```

### Debug session 2 — HTTP embedder & hybrid search wiring

**Problem:** Rust HTTP clients (ureq, reqwest, raw TcpStream) all hang on this system for unknown reasons (IPv4/IPv6 or proxy env). The only reliable approach is `curl` subprocess.

**Attempted approaches:**
1. `ureq` — `.set()` → `.header()` API mismatch, `into_json()` → `body_mut().read_json()`, hangs on POST
2. `reqwest` — hangs on POST
3. Raw `TcpStream::connect_timeout()` — leaves socket non-blocking; `set_nonblocking(false)` doesn't work on this kernel
4. `curl -d @file` — `@file` syntax doesn't work from Rust subprocesses
5. `curl -d @-` stdin pipe — works in standalone test but fails after ~42 sequential calls from Rust (pipe lifecycle issue)
6. **Final: `curl` via temp files + `sh -c` wrapper** — reliable across all 401 batches

**SGLang hang bug:** Certain long texts (~6.8K+ tokens with dense tokenization from ANSI escape codes or JSON arrays) cause SGLang's HTTP layer to hang without sending a response — the server processes inference but never writes the HTTP response back. Fixed with:
- 8K character truncation before embedding
- Empty-response detection as retryable

**Performance tuning (3,208 entries):**

| Batch size | Parallelism | Time |
|---|---|---|
| 8 | 1 (sequential) | ~14 min |
| 8 | 16 | ~4 min |
| 8 | 64 | ~60s |
| 16 | 64 | **~11s** |
| 64 | 64 | ~11s (plateau — SGLang per-batch inference dominates) |

**Final config:** batch_size=16, parallelism=64. Server handles sustained load without degradation.

**Results:** Hybrid search returns ranked results via RRF fusion over BM25 + vector scores. Top result for "test query" scores ~0.13 (vector-dominant).
