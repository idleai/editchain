### 3. Trajectory format
All agent trajectories are recorded as editchain operations (`Op`), using:
- `MessageOp` — system/user/assistant messages
- `ToolOp` — bash tool call lifecycle
- `CommandOp` — command execution output
- `NoteOp(Explains)` — exit status / submission metadata

### 4. Config
The config at `config/swebench.yaml` matches the target:
- Model: `deepseek-v4-flash` at `localhost:8000/v1`
- Temperature: 0.2, top_p: 0.95, max_tokens: 65536, frequency_penalty: 0.1
- Environment: Docker, cwd: `/testbed`, timeout: 60s
- Templates: system, instance, observation, format-error (Jinja2 via minijinja)

## Observations

1. **Agent loop works correctly** — all reached instances submitted successfully via the `COMPLETE_TASK_AND_SUBMIT_FINAL_OUTPUT` mechanism.

2. **Model server stability** — the sglang server at :8000 had intermittent connection errors under concurrent load (>2 workers). Running with 1–2 workers was stable but slow (~20–30 min per instance).

3. **Patch quality** — some patches failed to apply cleanly (malformed patch lines). This is likely due to the model producing patches with trailing whitespace issues or incomplete context lines.

4. **Trajectory recording** — all agent steps are recorded as editchain ops in `.traj.json` files, enabling replay and analysis.

5. **Target comparison** — the target config achieves ~70% on Verified; our result of **60.6%** is close but slightly below. The gap is likely attributable to:
   - Model server stability issues causing some instances to error out
   - Patch formatting differences between the Python-based mini-swe-agent and our Rust implementation
   - Template rendering differences (minijinja vs jinja2)
