## Executive decision

The current Claude Code importer is a useful prototype, but several correctness and storage issues should be fixed before a retrieval index is built on top of it.

The recommended design is:

1. **Make import lossless, streaming, and append-stable.** Give every physical session or subagent a deterministic source lane, use a collision-free operation ID layout, spill large records to blobs, persist cursors, and append bounded checksummed frames.
2. **Keep the edit chain as the only source of truth.** Text, vector, metadata, and graph indexes are rebuildable in-memory projections keyed by chain generation.
3. **Use Tantivy for BM25 and fielded search.** It is Rust-native, mature, segmented, and supports phrase queries, positions, filters, and custom code/path tokenizers.
4. **Use an exact mutable vector index first.** For a 150 MB-class corpus, normalized `f16` vectors and SIMD dot products may be sufficient. Add HNSW only after measured query latency justifies it.
5. **Fuse BM25 and dense results with Reciprocal Rank Fusion.** Rank fusion avoids adding incompatible raw BM25 and cosine scores.
6. **Run a warm per-chain daemon behind the CLI.** A daemon keeps the model, indexes, and graph hydrated while preserving a clean `editchain search|retrieve|summarize` interface.
7. **Publish generation watermarks.** Lexical and graph projections should become visible within hundreds of milliseconds; vector indexing may trail by a small microbatch and must report that lag.
8. **Exploit the DAG after textual candidate generation.** Use frontier filtering, causal expansion, branch diversity, occurrence collapse, and path explanations as bounded reranking/context operations.
9. **Keep summarization extractive in the first release.** Build deterministic, cited context packs rather than introducing semantic extraction or an embedded generative subsystem.

Recommended embedding bakeoff:

- **Qwen3-Embedding-0.6B at 512 dimensions:** quality-oriented default for mixed code and prose.
- **Nomic Embed Text v1.5 at 256 or 512 dimensions:** lighter operational baseline.
- **BGE-M3 at 1,024 dimensions:** heavier multilingual/long-context comparison, using dense output only.

## 1. Import and storage issues to fix first

The prior importer run processed five sessions and emitted 7,858 total raw-plus-normalized operations. Its architecture is directionally useful, but the following issues are release blockers for a larger chain.

### 1.1 Operation IDs overlap after 1,000 source records

The normalizer assigns raw records at `seq` and derived records at `seq * 1000 + 1`. This overlaps:

```text
first normalized child of source record 1 -> sequence 1001
raw source record 1001                    -> sequence 1001
```

Use an explicit source position with checked packing:

```rust
pub struct SourcePosition {
    pub record_ordinal: u64,
    pub derived_ordinal: u16, // 0 = raw record
}

impl SourcePosition {
    pub fn to_seq(self) -> Result<u64, IdError> {
        if self.record_ordinal > (u64::MAX >> 16) {
            return Err(IdError::Overflow);
        }
        Ok((self.record_ordinal << 16) | u64::from(self.derived_ordinal))
    }
}
```

This provides 65,535 derived operations per source record and makes overflow explicit.

### 1.2 Every session currently shares one workspace lane

The importer derives one `NodeId` from the workspace and uses a global sequence across files. That flattens concurrent sessions, makes IDs dependent on file order, and makes safe restart or append behavior difficult.

Use one deterministic lane per physical source stream:

```text
NodeId = trunc64(SHA-256(
  "editchain:source-stream:v1" ||
  workspace_identity ||
  source_schema ||
  stable_session_or_agent_identity
))
```

Use `boot` as the source generation. If a transcript is truncated or rewritten, increment `boot` and retain the old generation rather than reusing IDs.

### 1.3 Raw records above 4 KiB are truncated

The core already supports `Payload::Blob(BlobRef)`, but the current normalizer stores only the first 4,096 bytes of large records. Replace prefix truncation with content-addressed spill:

```rust
pub trait BlobSink {
    fn put(&mut self, bytes: &[u8]) -> Result<BlobRef, ImportError>;
}

fn payload_for(bytes: &[u8], blobs: &mut dyn BlobSink) -> Result<Payload, ImportError> {
    if bytes.len() <= INLINE_LIMIT {
        Ok(Payload::Inline(bytes.to_vec()))
    } else {
        Ok(Payload::Blob(blobs.put(bytes)?))
    }
}
```

A record must always round-trip byte-for-byte.

### 1.4 Import buffers the unread corpus and loses cursor state at exit

The current path reads the file tail into a `Vec`, clones each line, accumulates operations in memory, and builds one page containing all encoded operations. The CLI also uses a memory-only cursor store.

Replace it with:

```text
read one complete JSONL line
    -> hash and parse
    -> emit raw + normalized operations
    -> spill large payloads
    -> append to a bounded page builder
    -> commit frame and cursor checkpoint
```

Use `BufRead::read_until(b'\n', &mut line)` and retain only a partial final line.

### 1.5 The segment format needs explicit frame boundaries

The current page format has magic and records but no total frame length, record count, or implemented checksum. Introduce a new framed format while retaining EC02 read compatibility:

```text
magic              [4]
format_version     u16
header_len         u16
frame_len          u32
record_count       u32
page_sequence      u64
commit_generation  u64
flags              u32
header_crc32c      u32
records            ...
payload_crc32c     u32
```

Recovery scans complete frames and ignores only an incomplete or corrupt trailing frame. Segment access must be iterator-based rather than `fs::read` plus `Vec<Page>`.

### 1.6 The imported graph needs real causal edges

The core supports parent sets, but imported raw records currently have no provider-level parent, and normalized children mostly point only to their raw wrapper.

Resolve and project at least:

- source/provider predecessor and `parentUuid` edges;
- branch/fork origins;
- subagent spawn and parent relations;
- tool start → result/finish;
- command start → output → finish;
- reflection/compaction coverage frontiers;
- explicit merge or checkpoint edges.

Forward references can be held in a bounded pending table keyed by provider UUID. Relationships discovered later can be represented by additive edge/note operations without rewriting immutable source operations.

## 2. Streaming import and durability contract

### 2.1 Pipeline

```text
session files / live tails
        │
        ▼
stable source-lane discovery
        │
        ▼
streaming line reader
  offset + ordinal + BLAKE3 + partial-tail handling
        │
        ▼
loose envelope parser
        │
        ├── raw occurrence op, always
        └── normalized message/tool/command/file/reflection ops
        │
        ▼
content-addressed blob spill
        │
        ▼
bounded frame builder, target 1–4 MiB
        │
        ▼
durable frame commit + chain generation
        │
        ├── persistent cursor checkpoint
        └── projector notification
```

Bound batches by bytes as well as record count because tool output and snapshots can be large.

### 2.2 Cursor

```rust
pub struct SourceCursor {
    pub source_stream: SourceStream,
    pub generation: u32,
    pub byte_offset: u64,
    pub next_record_ordinal: u64,
    pub prefix_hash: [u8; 32],
    pub committed_chain_generation: u64,
    pub identity_version: u16,
}
```

Commit order:

1. Store referenced blobs.
2. Append and flush the operation frame.
3. Record the new chain generation.
4. Persist the cursor referencing that generation.
5. Notify graph, lexical, and vector projectors.

A crash after the frame commit but before the cursor update causes deterministic replay. Admission by stable `OpId` makes that replay idempotent.

On source mutation:

- matching prefix and larger size: import only the suffix;
- smaller size or changed verified prefix: increment source `boot` and start a new source generation;
- retain all previous imported facts.

### 2.3 Import acceptance tests

The importer should pass these before retrieval work is considered production-ready:

1. More than 100,000 source records with normalized children produce no ID collision.
2. 68 KiB and 1 MiB lines round-trip through `BlobRef` exactly.
3. Reimport of unchanged data appends zero new logical operations.
4. A one-line suffix append changes only that source lane tail.
5. Adding an earlier-sorting session does not change existing IDs.
6. Truncation at every frame byte boundary recovers all complete frames and no partial record.
7. Source rewrite creates a new `boot` without replacing old data.
8. Provider parents, tool pairs, and subagent links resolve independent of file discovery order.
9. Chain scan uses bounded memory and does not allocate the full corpus.

Initial memory target: **below 1 GiB peak RSS**, with **below 512 MiB** as the preferred target.

## 3. Query plane: rebuildable projections over the canonical log

Search state should remain disposable:

```text
                    append-only edit chain
                             │
                      CommittedBatch bus
                             │
          ┌──────────────────┼───────────────────┐
          ▼                  ▼                   ▼
   graph/metadata       Tantivy projector     embedding queue
      projector                                   │
          │                  │                    ▼
          │                  │              vector projector
          └──────────────────┼────────────────────┘
                             ▼
                    immutable QuerySnapshot
                       published via ArcSwap
                             │
                    search / retrieve / summarize
```

Suggested crate split:

```text
editchain-query   request/response types, fusion, graph algorithms
editchain-index   text projection, Tantivy, metadata, snapshots
editchain-embed   model manifests and inference backends
editchain-node    append owner, commit bus, daemon
```

Readers load one immutable snapshot for an entire query:

```rust
pub struct QuerySnapshot {
    pub hydrated_generation: Generation,
    pub graph_generation: Generation,
    pub lexical_generation: Generation,
    pub vector_generation: Generation,
    pub lexical: Arc<LexicalSnapshot>,
    pub vectors: Arc<dyn VectorSnapshot>,
    pub graph: Arc<GraphSnapshot>,
    pub metadata: Arc<MetadataSnapshot>,
}
```

`ArcSwap<Arc<QuerySnapshot>>` provides atomic publication without a coarse reader lock.

## 4. Search document projection

Index two levels:

1. **Operation document:** one per searchable operation with metadata and compact payload text.
2. **Chunk document:** deterministic chunks for long messages, tool results, commands, diffs, and file content.

```rust
pub struct ChunkRecord {
    pub chunk_id: ChunkId,
    pub op_id: OpId,
    pub chunk_ordinal: u32,
    pub byte_start: u32,
    pub byte_end: u32,
    pub token_start: u32,
    pub token_end: u32,
    pub generation: Generation,
}
```

### Chunking policy

No semantic extraction is needed. Split only on existing structure:

1. Keep short operations intact.
2. Prefer message blocks, tool/result boundaries, command output boundaries, and diff hunks.
3. Apply a tokenizer-based sliding window to remaining long text.

Starting profile:

```text
window:   768 tokens
overlap:   96 tokens
hard cap: 64–128 KiB projected text per chunk
```

### Raw and duplicate policy

- Index normalized message/tool/command/file/reflection text by default.
- Index raw records only when normalized text is unavailable or `--include raw` is requested.
- Exclude private/thinking content unless explicitly enabled.
- Group copied physical occurrences by provider UUID/content hash while preserving occurrence metadata.
- Collapse multiple chunks from one operation in default output.

### Core fields

| Field | Purpose |
|---|---|
| `body`, `body_code` | prose and code-aware text |
| `path_text`, `path_exact` | path search and exact filtering |
| `tool_name`, `kind`, `tags`, `role` | structured filtering |
| `actor`, `session`, `node`, `boot`, `seq` | scope and lane filtering |
| `clock_ms`, `generation` | time and consistency filtering |
| `op_id`, `chunk_id`, `source_uuid` | identity and occurrence grouping |

Use separate prose, code-identifier, path, and exact-keyword tokenizers.

## 5. Hybrid retrieval

### 5.1 Lexical index: Tantivy

Tantivy is the recommended BM25 engine because it is Rust-native, fielded, position-aware, and segment-based. Start with `RamDirectory` and a near-real-time commit loop:

```text
max interval:      100 ms
max uncommitted:   1,000 documents or a byte threshold
commit -> reader reload -> publish QuerySnapshot
```

If commit/reload p99 or segment churn becomes a problem, add a small mutable lexical delta in front of committed Tantivy segments. Do this only after measurement.

### 5.2 Dense index: exact first

Normalize vectors at insertion so cosine similarity becomes a dot product. Store row-major `f16` vectors in one active segment plus immutable sealed segments:

```rust
pub struct FlatVectorSegment {
    pub generation_start: Generation,
    pub generation_end: Generation,
    pub dimensions: usize,
    pub doc_ordinals: Vec<DocOrdinal>,
    pub vectors: AlignedVec<f16>,
}
```

Approximate vector memory:

| Vectors | 512-d `f16` |
|---:|---:|
| 100,000 | 97.7 MiB |
| 200,000 | 195.3 MiB |
| 500,000 | 488.3 MiB |

Maintain `RoaringBitmap` filters for kind, session, lane, tags, path, visibility, and generation. A filtered flat scan can be substantially cheaper than scanning every vector.

Add HNSW only when measured vector p95, CPU utilization, or corpus size exceeds the flat-index budget. Preserve an exact active delta and use ANN only for sealed segments or a compacted base. Keep this behind a `VectorIndex` trait; USearch is the most feature-rich later candidate, while `hnsw_rs` avoids a C++ dependency.

### 5.3 Rank fusion

Starting candidate sizes:

```text
BM25 candidates:   200
vector candidates: 200
final results:       20
```

Use Reciprocal Rank Fusion:

```text
RRF(d) = Σ 1 / (60 + rank_r(d))
```

Then:

1. exact-rerank ANN candidates when applicable;
2. collapse duplicate chunks and copied occurrences;
3. apply a bounded graph/branch diversity adjustment;
4. use deterministic ID tie-breaking.

Keep raw BM25 and cosine scores in debug output, but do not add them directly.

### 5.4 Fresh records

A newly hydrated record remains searchable through BM25 even before its vector is ready. Each response reports:

```json
{
  "document_generation": 4812,
  "lexical_generation": 4812,
  "vector_generation": 4807,
  "vector_ready": false
}
```

Strict hybrid queries may wait for vector coverage through a requested generation.

## 6. Online consistency and daemon model

A warm process is necessary to keep indexes, tokenizers, graph state, and the embedding model ready:

```bash
editchain serve --chain .editchain --watch
```

CLI commands connect over a Unix-domain socket. Use one append coordinator per chain directory so all local writers share one ordered physical log and monotonic commit generation.

### Watermarks

```rust
pub struct ProjectionWatermarks {
    pub log: Generation,
    pub hydrated: Generation,
    pub graph: Generation,
    pub lexical: Generation,
    pub vector: Generation,
}
```

Consistency modes:

```bash
# Current graph and lexical snapshot; vector lag is reported.
editchain search "..." --consistency lexical

# Wait for full hybrid coverage.
editchain search "..." --consistency hybrid --wait 750ms

# Read-your-writes after append generation 4812.
editchain search "..." --min-generation 4812 --wait 1s
```

Initial warm-process targets:

| Event | Target |
|---|---:|
| durable append acknowledgement | p99 < 100 ms |
| graph visibility | p99 < 100 ms |
| BM25 visibility | p99 < 250 ms |
| vector visibility | p99 < 1 s |
| BM25 top-20 query | p95 < 50 ms |
| hybrid top-20 query | p95 < 150 ms |

Use separate bounded queues. Graph and lexical projection have priority; embedding may lag. Never allow an unbounded embedding queue.

## 7. DAG-aware retrieval

Apply graph algorithms only to strong text/vector candidates and bounded neighborhoods.

### Frontier-aware filtering

For a frontier mapping `(node, boot) -> max_seq`:

```text
visible(op, frontier) := op.seq <= frontier[op.node, op.boot]
```

This gives fast “what was known at this checkpoint?” queries without traversing the whole graph.

### Causal cone expansion

For each selected hit, retrieve a bounded context neighborhood:

```text
ancestors:       2
 descendants:    1
nodes per seed: 128
maximum total:  512
```

Ancestors recover the prompt, tool start, command invocation, and branch origin; descendants recover results, file effects, tests, and conclusions.

### Causal corridor

For `retrieve --why`, compute a shortest or minimum-cost parent path from a hit to a selected tip/frontier. Return the ordered operation IDs and snippets that connect the evidence to later work.

### Branch diversity

Use a graph-aware MMR penalty based on same operation, same logical UUID, same session/lane, ancestor-cone overlap, and graph distance:

```text
utility = relevance
        - λ_text  * duplicate_text_similarity
        - λ_graph * graph_overlap
```

This avoids returning twenty nearly identical chunks from one branch.

### Occurrence collapse

Maintain:

```text
logical provider/content key -> physical source occurrences
```

Default output shows one logical result plus occurrence count and branch/session locations. An occurrence mode expands each physical record.

### Later optimizations

- Seeded relevance diffusion on a small induced subgraph.
- Stable-spine/dominator analysis for branch-independent summaries.
- Graph-relative recency based on distance to active tips.
- Dynamic reachability labels such as DAGGER or O’Reach only if bounded BFS and lane frontiers become a bottleneck on multi-million-node DAGs.

## 8. CLI surface

Keep stdout machine-stable and send diagnostics to stderr. Every query command supports `--json` or `--jsonl`.

### Lifecycle

```bash
editchain serve --chain .editchain --watch \
  --lexical-commit-interval 100ms \
  --embedding-model Qwen/Qwen3-Embedding-0.6B \
  --embedding-dim 512

editchain index build --chain .editchain --rebuild
editchain index status --chain .editchain
editchain doctor corpus --sessions-dir <sessions-dir>
editchain doctor chain --chain .editchain
```

### Search

```bash
editchain search "mismatched a_input shape" \
  --mode hybrid \
  --top 20 \
  --path 'flashinfer/**' \
  --kind message,tool,command \
  --ancestors 2 \
  --descendants 1 \
  --group-by logical \
  --consistency lexical
```

Important filters:

```text
--session --actor --node --kind --tag --path
--after --before --at-frontier --visible-from
--include raw --include-private
--min-generation --wait
```

### Retrieve

```bash
editchain retrieve --op <op-id> --raw
editchain retrieve --chunk <chunk-id> --context 2:2
editchain retrieve --op <op-id> --ancestors 3 --descendants 2
editchain retrieve --op <op-id> --why-tip <tip-op-id>
```

### Summarize

The first implementation should be deterministic and extractive:

```bash
editchain summarize \
  --query "why the static mxfp4 kernel recompiles per M" \
  --budget-tokens 6000 \
  --strategy context-pack \
  --include-graph-corridors \
  --format markdown
```

Strategies:

- `extractive`: top distinct evidence snippets;
- `timeline`: causal/time-ordered excerpts;
- `context-pack`: stable spine, branch-diverse evidence, and recent unresolved work;
- `diff-focus`: messages and commands around matching file operations.

Every item includes `OpId`, source occurrence, byte/token range, and generation.

### Tail and read-your-writes

```bash
editchain tail --follow --since-generation 4800
GEN=$(editchain append --json event.json --print-generation)
editchain search "new fact" --min-generation "$GEN" --wait 1s
```

## 9. Open-weight embedding choices

| Model | Dimensions/context | Best fit | Main trade-off |
|---|---|---|---|
| **Qwen3-Embedding-0.6B** | selectable up to 1,024; 32K context | mixed prose, code, multilingual retrieval | heavier inference |
| **Nomic Embed Text v1.5** | 768 base; Matryoshka 512/256/128/64; 8K context | compact CPU-oriented baseline | weaker multilingual/code emphasis |
| **BGE-M3** | 1,024; 8K context | long and multilingual comparison | larger vectors and more compute |

Recommended first configurations:

```text
Qwen3-Embedding-0.6B: 512 dimensions
Nomic v1.5:            256 and 512 dimensions
BGE-M3:              1,024 dimensions, dense output only
```

For Qwen, pin a query instruction. For Nomic, pin the required `search_query:` and `search_document:` prefixes.

### Inference backend

Use an `Embedder` trait and start with **FastEmbed-rs** for in-process Rust inference. Keep **Text Embeddings Inference** as an optional sidecar when GPU batching or model sharing across chains is desirable.

Every vector snapshot must pin:

```rust
pub struct EmbeddingManifest {
    pub model_id: String,
    pub revision: String,
    pub tokenizer_revision: String,
    pub backend: String,
    pub dimensions: u32,
    pub max_tokens: u32,
    pub pooling: Pooling,
    pub normalize: bool,
    pub query_instruction: String,
    pub document_prefix: String,
    pub chunk_profile: String,
}
```

Any manifest change forces a vector rebuild. Never mix vectors produced by different model revisions, prefixes, dimensions, or pooling rules.

Evaluate BM25 and each hybrid configuration on a judged set covering exact errors, paths and symbols, paraphrased causes, tool output, branch-local facts, copied prefixes, and newly appended subagent work.

## 10. Implementation sequence

### PR A — import correctness

- deterministic source lane per session/subagent;
- checked source-position ID allocation;
- large-record blob preservation;
- collision, order-stability, and byte-roundtrip tests.

### PR B — streaming durable log

- streaming JSONL iterator;
- persistent cursor store;
- bounded page sink;
- EC03 frame length/checksum;
- streaming segment iterator;
- restart and partial-write recovery tests.

### PR C — lexical and graph projection

- deterministic chunker and manifest;
- dense ordinals and roaring metadata filters;
- Tantivy BM25 with code/path tokenizers;
- graph adjacency and frontier projection;
- `search --mode lexical`, `retrieve`, and extractive `summarize`.

### PR D — daemon and near-real-time publication

- single append coordinator and commit generations;
- projector bus;
- Tantivy microcommit/reload;
- `ArcSwap` snapshots;
- Unix socket, watermarks, wait semantics, backpressure, and metrics.

### PR E — vector and hybrid

- `Embedder` and model manifest;
- exact `f16` vector segments;
- filtered scan;
- RRF, exact reranking, and occurrence collapse;
- model bakeoff and retrieval-quality harness.

### PR F — graph retrieval

- frontier filtering;
- causal cone and corridor retrieval;
- branch-aware diversity;
- context-pack summarization;
- optional ANN and reachability accelerators only when benchmarks require them.

## 11. Initial defaults and acceptance targets

```text
inline payload threshold:       4 KiB, spill rather than truncate
frame target:                   2 MiB
segment rotation:             128 MiB
Tantivy commit interval:      100 ms
max uncommitted docs:        1,000
chunk window / overlap:     768 / 96 tokens
embedding dimensions:          512
embedding batch wait:           25 ms
embedding batch documents:      32
BM25 candidates:               200
vector candidates:             200
RRF k:                          60
final results:                  20
graph expansion:                 2 ancestors / 1 descendant
per-seed graph budget:          128 nodes
private content indexing:       off
raw JSON search:                off by default
result grouping:                logical occurrence collapse
```

Measure:

- import throughput and peak RSS;
- byte-perfect source recovery and zero invalid duplicate IDs;
- p50/p95/p99 lexical, vector, hybrid, and graph-expanded queries;
- append-to-BM25 and append-to-vector visibility;
- retrieval Recall@20, nDCG@10, and MRR;
- incremental projection versus full rebuild equivalence;
- behavior under parallel writers, queries, embedding, and compaction.

A nonzero invalid-duplicate quarantine during deterministic import is a correctness failure.

## Final recommendation

Build the first release around this minimal stack:

```text
lossless streaming importer
        +
checksummed append-only frames with generations
        +
Tantivy BM25 in RAM
        +
roaring metadata and frontier filters
        +
exact 512-d f16 vector scan
        +
Reciprocal Rank Fusion
        +
bounded DAG-aware expansion and diversity
        +
warm Unix-socket daemon behind the CLI
```

This is sufficient for the first 150 MB-class history workload and has the right seams for substantially longer DAGs. The immediate priority is import correctness and online projection, not ANN tuning. HNSW, dynamic reachability labels, MCP, semantic extraction, and generative summarization can remain later additions.
