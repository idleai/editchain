## 1. Objective

Build a rich, keyboard-first terminal interface for exploring EditChain data.

The TUI must support two complementary visualization modes:

1. **Full-chain DAG log**
   - Git-log-style lane rendering
   - One operation per row
   - Branches, merges, and causal relationships
   - Scales to large chains through row virtualization

2. **Full individual-node inspection**
   - Every envelope field
   - Kind-specific properties
   - Decoded payloads
   - Parents, children, and related operations
   - Raw JSON and storage metadata

The intended first command is:

```bash
editchain-tui ./outputs/my-chain
```

A later integration may expose the same tool through:

```bash
editchain tui ./outputs/my-chain
```

---

## 2. Product Direction

Use a terminal-native stack instead of a browser or desktop GUI.

### Recommended stack

```text
Ratatui
Crossterm
Petgraph
tui-input
tui-scrollview
tui-markdown
ansi-to-tui
Tokio or a dedicated worker thread
```

### Why this direction fits EditChain

- EditChain is naturally navigated as a chronological and causal log.
- A terminal lane view can display branches and merges compactly.
- Large chains can be explored without drawing every node on a graphical canvas.
- The TUI can connect directly to the Rust data model.
- No web frontend, JavaScript framework, browser server, or desktop packaging layer is required.
- The result is easy to run over SSH and inside developer environments.

## 6. Storage and Loading Layer

The viewer must be read-only.

Opening a path must:

- require an existing chain directory
- never create a directory
- never rotate segments
- never append data
- tolerate partial trailing records
- detect supported frame formats
- report corrupt records without crashing the entire application

### Required reader API

```rust
pub struct ChainReader {
    // Existing chain directory and segment metadata.
}

pub struct RecordLocation {
    pub segment: u32,
    pub frame_offset: u64,
    pub record_offset: u64,
    pub record_len: u32,
    pub format: FrameFormat,
}

pub struct ScannedRecord<'a> {
    pub location: RecordLocation,
    pub flags: u32,
    pub bytes: &'a [u8],
}

impl ChainReader {
    pub fn open_existing(
        path: impl AsRef<std::path::Path>,
    ) -> std::io::Result<Self>;

    pub fn scan(
        &self,
        visitor: impl FnMut(ScannedRecord<'_>) -> Result<(), ScanError>,
    ) -> Result<ScanReport, ScanError>;

    pub fn read_record(
        &self,
        location: RecordLocation,
    ) -> Result<Vec<u8>, ScanError>;
}
```

### Loading strategy

Do not decode and retain every complete operation payload.

During initial scanning:

1. Read one segment or bounded block at a time.
2. Decode the operation envelope.
3. Build a compact `OpHeader`.
4. Store the record location.
5. Retain only a short preview where useful.
6. Build lookup and filtering indexes.
7. Emit progress events.
8. Release raw segment buffers.

Decode the complete operation only when the inspector requests it.

## 11. Lane Rendering Algorithm

Implement the DAG lane column as a custom Ratatui widget.

### Data structures

```rust
pub struct DagRow {
    pub op: OpOrdinal,
    pub cells: Vec<LaneCell>,
    pub transition: Option<Vec<LaneCell>>,
}

pub enum LaneCell {
    Empty,
    Vertical,
    Node,
    SelectedNode,
    Horizontal,
    BranchLeft,
    BranchRight,
    MergeLeft,
    MergeRight,
    Crossing,
    ManyParents,
}
```

### Lane state

```rust
pub struct LaneState {
    pub active: Vec<Option<OpId>>,
}
```

### Processing order

Process operations newest-first for Git-style parent expansion.

For each operation:

1. Find the lane already associated with the operation.
2. If absent, allocate a lane.
3. Draw the operation marker in that lane.
4. Remove the operation from the active lane.
5. Insert its parent IDs into active lanes.
6. For two parents:
   - keep one parent in the current lane
   - allocate or reuse another lane for the second
7. If another lane already represents a parent:
   - draw a merge
   - collapse duplicate lanes
8. Emit a transition row when lines cross.
9. For unresolved `ParentSet::Many`:
   - display a special marker such as `⋯`
   - show the blob reference in the inspector

### Determinism

Use stable ordering for ties:

```text
clock value
clock sub-counter
node ID
boot
sequence
```

The renderer should produce the same rows for the same visible operation set.

### Filtering behavior

Two modes may eventually be supported:

1. **Structural filtering**
   - hidden operations disappear
   - graph lines may become disconnected

2. **Context-preserving filtering**
   - matched operations are emphasized
   - connector operations remain dimmed

For the MVP, start with structural filtering and clearly indicate hidden-parent gaps.

## 12. Virtualized Rows

The DAG log must draw only rows visible in the terminal viewport.

Use a fixed row height:

```text
1 terminal row per operation
```

Do not wrap previews.

Long content is clipped in the list and shown fully in the inspector.

The visible operation order should be precomputed whenever:

- the chain loads
- filters change
- sorting changes
- a search mode changes

Do not recompute bitmap intersections inside every render call.

## 18. Local Two-Dimensional Graph

Add this after the DAG log is stable.

The local graph is bounded around the selected operation.

### Default request

```text
ancestors:   2
descendants: 1
maximum nodes: 100
```

### Graph content

Primary edges:

- causal parent edges

Optional overlay edges:

- note relationships
- tool lifecycle
- command lifecycle
- file revision
- normalized-to-import derivation

Only causal parent edges should constrain layout.

### Layout

Use a deterministic layered layout:

1. Build the bounded neighborhood.
2. Detect cycles.
3. Topologically sort causal edges.
4. Assign rank from parents to children.
5. Order nodes within ranks.
6. Run a few crossing-reduction sweeps.
7. Assign terminal coordinates.
8. Route edges with simple orthogonal segments.
9. Draw secondary relationships with distinct line styles.

### Rendering

Use Ratatui’s canvas or a custom buffer widget.

The local graph should support:

- pan
- keyboard node selection
- zoom or detail-level toggling
- jump back to DAG log
- open selected graph node in the inspector

Never render the complete chain in this view.

## 20. UI Event Loop

A simplified loop:

```rust
loop {
    terminal.draw(|frame| app.render(frame))?;

    while let Some(event) = backend.try_recv() {
        app.handle_backend_event(event);
    }

    if crossterm::event::poll(tick_rate)? {
        let event = crossterm::event::read()?;
        app.handle_terminal_event(event);
    }

    app.tick();

    if app.should_quit {
        break;
    }
}
```

If Tokio event streams are used, merge:

- keyboard events
- resize events
- backend events
- periodic ticks
- live tail notifications

into one asynchronous event loop.

## 21. Error Handling

The TUI should recover from local failures wherever possible.

### Recoverable warnings

- operation decode failure
- unresolved parent
- missing blob
- partial trailing record
- invalid UTF-8
- unknown operation kind
- unsupported payload format
- graph neighborhood truncation

Show warnings in:

- the status bar
- a diagnostics popup
- the selected operation inspector where relevant

### Fatal errors

- chain path does not exist
- chain path is not readable
- no supported segment format
- terminal initialization failure

Fatal errors should restore the terminal before printing diagnostics.

Use a panic hook that exits alternate-screen mode and restores cursor visibility.

## 22. Privacy and Safety

Defaults:

```text
raw imports hidden
private operations hidden
blob payloads not loaded automatically
read-only chain access
no external network calls
```

When private records are enabled:

- show a persistent visible indicator
- do not persist the setting by default
- preserve filtering in search
- avoid copying private content unless explicitly requested

Imported content must be treated as data.

Never:

- execute embedded commands
- emit raw control sequences
- render arbitrary terminal escape codes directly
- follow external links automatically

## 26. Acceptance Criteria

The initial release is complete when:

1. The application opens an existing EditChain directory read-only.
2. It displays at least 130,000 operations without constructing one widget per operation.
3. The default view shows a recognizable causal lane graph.
4. Branches and two-parent merges render deterministically.
5. Every operation can be selected.
6. Every envelope property is visible.
7. Every `OpKind` has an appropriate content renderer.
8. Parents and children can be followed from the keyboard.
9. Search and filters work without blocking rendering.
10. Raw imports are hidden by default.
11. Private records are hidden by default.
12. Full payloads are decoded lazily.
13. Partial or corrupt records produce diagnostics rather than crashes.
14. The terminal is restored after normal exit, errors, and panics.
15. The bounded graph view never attempts to render the complete chain.
16. The application makes no external network requests.
17. Opening an invalid path creates no files or directories.

## 27. Main Risks and Mitigations

### Risk: DAG lanes become complex

Mitigation:

- start with roots, linear chains, branches, and two-parent merges
- use golden text fixtures
- treat `Many` as unresolved initially
- isolate lane generation from Ratatui rendering

### Risk: Large chains consume too much memory

Mitigation:

- streaming scanner
- compact headers
- lazy detail decoding
- capped previews
- record-location lookup
- LRU detail cache

### Risk: Terminal layouts become crowded

Mitigation:

- responsive column hiding
- fixed-height rows
- full-screen node mode
- horizontal scrolling only where necessary
- context-sensitive help

### Risk: Imported terminal output is unsafe

Mitigation:

- parse ANSI into styled text
- reject unsupported control sequences
- never print raw imported bytes directly
- hex-render binary data

### Risk: Filtering breaks causal readability

Mitigation:

- show missing-parent indicators
- later add context-preserving filtering
- provide a local neighborhood view around matches

### Risk: Storage code remains coupled to the CLI

Mitigation:

- extract `editchain-store`
- keep readers and writers separate
- keep TUI dependencies out of `editchain-node`

## 29. Final Technical Decision

Use:

```text
Ratatui + Crossterm
Git-style DAG lane view as the primary visualization
Petgraph for traversal and bounded neighborhoods
Full tabbed node inspector
Streaming read-only chain scanner
Compact immutable snapshot
Lazy payload decoding
Background worker for loading, filtering, and search
Bounded local canvas as a secondary visualization
```

The key product model is:

> The complete EditChain is a navigable causal log. The selected operation receives the full rich visualization.

This is the smallest practical path to a powerful EditChain inspection experience while remaining entirely within the Rust and terminal ecosystem.
