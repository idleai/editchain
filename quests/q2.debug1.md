## Bug Found & Fixed: UTF-8 Boundary in Chunked Text Indexing

**Location**: `crates/editchain-index/src/lexical.rs:138`

**Symptoms**: `editchain search` panicked with:
```
start byte index 10752 is not a char boundary; it is inside '—' (bytes 10751..10754)
```

**Root cause**: The chunker (`chunk_text`) computes byte offsets using a rough token estimate (4 bytes/token), which can land in the middle of multi-byte UTF-8 characters (e.g., em-dash `—` is 3 bytes). When the lexical indexer slices `&text[start..end]` using these offsets, it hits non-char boundaries.

**Fix**: Added `floor_char_boundary()` calls to adjust both start and end offsets to valid UTF-8 boundaries before slicing:

```rust
let start = text.floor_char_boundary(start);
let end = text.floor_char_boundary(end);
let ct = &text[start..end];
```

Also fixed a Rust precedence issue (`as usize.min()` → `(as usize).min()`).

**Also fixed**: The search results had empty `text` fields because the stored Tantivy body field wasn't being extracted from the document. Added:
```rust
let body_text = doc.get_first(self.fields.body)
    .and_then(|v| v.as_str())
    .unwrap_or("")
    .to_string();
```

## Practical Retrieval Use Cases

### Use Case 1: MXFP4→NVFP4 conversion strategy

```
editchain search "mxfp4 nvfp4 conversion" --mode lexical --top 10
```

Found the exact git commit messages, conversion commands, and analysis messages documenting the closed-form scale remapping approach. Top result was the commit adding `QUEST_MXFP4_TO_NVFP4_W4A16.md` with the full 8-iteration debug journey.

### Use Case 2: FlashInfer W4A16 kernel dequant conventions

```
editchain search "flashinfer kernel w4a16 dequant" --mode lexical --top 10
```

Returned detailed analysis of FlashInfer's `moe_w4a16_kernel.py` — the `packed_dequant_e2m1x4_to_bfloat2x2` SMEM dequant path, scale processing pipeline (`block_scale * global_scale = 2^k_j`), and the `prepare_w4a16_packed_weights` format expectations.

### Use Case 3: SGLang MXFP4 MoE backend routing

```
editchain search "sglang backend mxfp4 w4a4 moe" --mode lexical --top 10
```

Found the complete backend sweep results table showing which MoE backends work on SM120, the `mxfp4_w4a4_moe.py` apply() routing logic, and the environment variables controlling backend selection.

### Use Case 4: CUDA graph static kernel JIT compilation

```
editchain search "cuda graph static kernel jit compilation" --mode lexical --top 10
```

Returned deep analysis of the JIT compilation root cause — the static kernel cache key includes M (token count), causing ~23s per unique M compilation. Found the `_StaticMoELaunch` runtime-m wrapper fix and the startup phase timing breakdown (210s total).

### Use Case 5: Blackwell SM120 tensor core FP4 throughput


Found the comprehensive FP4 MoE backend sweep results table comparing marlin GEMV (526K tok/10min) vs FP8 Triton MoE (804K tok/10min), plus deep research prompts about the checkpoint format gap between MXFP4 and ModelOpt FP4.

## Performance Observations

- **Index build time**: ~2 minutes for 129K operations (Tantivy BM25 in RamDirectory)
- **Search latency**: ~2 seconds for BM25 top-10 on full corpus (cold start, no daemon)
- **Summarize latency**: ~2 seconds (search + snippet extraction)
- **Retrieve latency**: ~1 second (linear scan by op ID)
- **Chain dump**: ~5 seconds for full JSONL output

The current implementation rebuilds the Tantivy index from scratch on every command invocation. With a warm daemon (`editchain serve`), search latency would drop to <50ms p95.
